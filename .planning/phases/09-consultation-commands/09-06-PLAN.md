<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args (AMOUNT, --note) → SQL | Both pass through bindparams; AMOUNT is float, --note is str |
| User TTY input (y/n) → state change | confirm_prompt is the sole gate before UPSERT |
| Engine → backups/ directory | create_backup writes file using sqlite3 backup API |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-06-T1 | Tampering | AMOUNT injection | mitigate | Typer parses as `float`; UPSERT uses bindparam — no string interpolation |
| T-09-06-T2 | Tampering | --note injection in audit payload | mitigate | Stored via `json.dumps` in log_event; on display in `audit` command, parsed via json.loads (Plan 09-05) |
| T-09-06-T3 | Tampering | UPSERT race vs init | mitigate | Single-user CLI; `on_conflict_do_update` is atomic at SQLite layer |
| T-09-06-I1 | Information Disclosure | cash value to terminal | accept | Single-user CLI; user reading own data |
| T-09-06-R1 | Repudiation | cash change without audit | mitigate | log_event(CASH_UPDATED) is in the same code path as the UPSERT and create_backup — backup happens between write and log; if log_event fails the next scan will note the discrepancy. Single-user trust model; ASVS L1 satisfied. |
| T-09-06-V1 | Tampering | negative amount | mitigate | `amount < 0.0` rejected with print_error + exit code 1 |
| T-09-06-SC | Tampering | npm/pip/cargo installs | accept | No new packages |
</threat_model>
