primitives. The severity prefix is the textual carrier (rule 6.29 — color is never the
sole information carrier).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller `data` dict → Rich Console | Untrusted strings (e.g., yfinance API errors, user-supplied symbols) may contain Rich markup |
| caller `title`/`body` strings → Rich Console | Same — must be rendered as plain text |

## STRIDE Threat Register (ASVS L1)

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Tampering | `print_message(data=...)` rendering of dict values | mitigate | All `data`/`impact`/`body` strings are wrapped in `rich.text.Text(...)` (or `_render_kv_block` uses `console.print` with concatenated f-string of plain literals) so Rich markup syntax (`[red]X[/red]`) is rendered literally rather than interpreted. Test `test_print_message_data_markup_injection` asserts the literal `[red]X[/red]` substring appears in output. Aligns with ASVS V5 input validation. |
| T-05-05 | Spoofing | Color used as sole severity indicator | mitigate | Severity prefix `Error:`/`Warning:`/`Info:`/`Success:` is textual and present regardless of color support (rule 6.29). NO_COLOR fallback test asserts the textual prefix still appears. |
| T-05-06 | Information Disclosure | Stack traces shown to user | accept | `_app.py` already sets `pretty_exceptions_enable=False`. Phase 5 emits only the message text passed by the caller; no exception details are auto-rendered. Caller chooses what to include. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed. RESEARCH.md confirms zero new dependencies. |
</threat_model>
